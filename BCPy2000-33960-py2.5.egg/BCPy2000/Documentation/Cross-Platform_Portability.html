Cross-Platform Portability

Much of the BCI2000 source distribution can already compile under gcc, but currently (April 2009) anything with GUI or graphical elements, including the Operator, still relies on Borland's proprietary VCL library, and hence also on Windows. This means that platform-independence for the whole BCI2000 project is still some time away.

However, early porting of the Python-based modules to other platforms has been made relatively easy by the portable nature of Python itself. We have had some success in running them on GNU/Linux and Mac OS X, although a Windows machine is still required for the Operator. Soon after the essential functions of the Operator module are separated from its UI functions (one of the project's milestones for 2009) it should be possible, in principle, to run entirely Windows-free (although note that the drivers for most commercial EEG signal acquisition systems, and hence also most Signal Source modules, will probably remain Windows-dependent for the foreseeable future).

As of the 11336 release (March 2009), the gbuild bash scripts can be used to build BCPy2000's three foundation binaries with gcc. On Windows, this means using cygwin, and the -mno-cygwin flag is passed to gcc (tested with gcc 3.4.4).

Here are our experiences with other platforms so far. We have not yet performed any testing beyond an initial run to prove that the demos work. Please let us know (using the support e-mail address) if you manage to get any further.

GNU/Linux

On Ubuntu hardy running on i686 and compiling with gcc 4.2.3, everything seemed to run, except for sound stimuli since I haven't managed to get PyAudio working on that distro. VisionEgg failed to operate at anything above 16 bits per pixel, and rendering was very slow - presumably due to suboptimal graphics setup on my particular system.

Mac OS X

Under Mac OSX (10.5) on i386 and compiling with gcc 4.0.1, likewise all modules run. Getting VisionEgg working at all on OSX is a challenge, but once this works, BCPy2000 does too.

On Windows and Linux, the -DDYNAMIC_PYTHON=1 flag may be used, allowing the binary to be compiled independent of any particular Python version so that the desired Python shared library and distribution can be chosen via a command-line switch when the executable is launched. On OSX, however, the result gave an "Illegal Instruction" message when run. Only executables compiled with -DDYNAMIC_PYTHON=0 and statically linked to a particular Python library have worked for me on OSX so far.
