Renderers

BCPy2000 still loves VisionEgg, but it's an open relationship.

All pygame- and VisionEgg-specific code is now in the file VisionEggRenderer.py, which defines a backend for drawing visual stimuli and responding to UI events. The class VisionEggRenderer is defined as a subclass of BciGenericRenderer, and is registered as the subclass to use by being attached as the subclass attribute of the BciGenericRenderer class object.

If, however, you write a similar file with an analogous implementation, and simply import it in the main part of your BciApplication file, then your implementation will replace ours.

The renderer instance is available as self.screen or from non-object-oriented code using 

from AppTools.CurrentRenderer import screen

The new self.screen behaves much as the old, in the sense of having bgcolor and size properties. It has one additional attribute if you're using the tools: if you import AppTools.Displays and use init_screen or fullscreen, then the screen coordinates are available at self.screen.coords. This is the AppTools.Boxes.box object that used to be available from default_coordinate_frame in the now-abolished AppTools.ScreenCoordinates module.

The process of writing a renderer module is documented to some extent in the BciGenericRenderer and BciStimulus class docstrings. Alternative renderers based on pygame alone (no OpenGL dependency) and on pure console input/output, are under construction for the next release, for use as templates.
