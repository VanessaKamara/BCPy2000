Known Issues

If anyone out there has tips for dealing with these, please let us know using the support e-mail address.

  • Timing
    • Video frames are occasionally dropped. So far we have not been able to eliminate framedrops entirely.
    • BCI2000 timing does not work smoothly on virtual machines. Your mileage may also vary considerably as regards the performance of OpenGL and VisionEgg. The BCPy2000 demos have worked for me in a Virtual Box, but were slow, jerky and ugly. The application module has got particularly slow (using 100% CPU all the time and slowing everything down to a crawl) as of release 11336, but this doesn't seem to be a problem on non-virtual win32 machines.
    • Our EventOffset mechanism seems to work with reasonably high precision, but still sometimes low accuracy: sound recording tests show a bias that seems to be parameter-dependent. It is constant for a given parameter setting, however, so it's not much of a problem for BCI purposes (just turns P300s consistently into, say, P350s). In any case, note that a physical synchronization signal is always a much better option.
  • Shell
    • Pressing ctrl-C at the shell prompt causes the main thread to be interrupted and the whole module breaks. This is by design in IPython, but it would be nice to prevent it since it's sometimes useful to be able to send a ctrl-C just to the shell environment itself (to get out of stupid multi-line bracket-nesting traps for example). I just don't know how to disable this feature yet.
    • Shells do not save their histories when they quit. Adding a self._ipshell.IP.savehist() call seems to work, but only very intermittently (perhaps due to race conditions when more than one module is running a shell?)
  • Debugger
    • Whenever the pdb debugger (launched by self.dbstop) quits, it throws an exception of type Bdb (presumably by design). It's not clear where to catch this without hacking the debugger's own source, so currently the debugger is a one-shot deal: you can go into it, but never out of it without restarting the whole system. You can, however, issue the c command for 'continue'.
    • When the debugger opens, it and the shell prompt are running in parallel threads and they are both listening to your keyboard strokes. I've performed a wild hack in order to prevent this, but the hack is only effective from the second line onwards. Therefore it's a good idea to press 'return' once at the beginning of a debugger session - IPython should then suspend itself and let you debug in peace (although it seems to come back very occasionally, under circumstances that are not quite clear...)
  • Non-Windows
    • On non-win32 platforms only: when a shell and a log file are used simultaneously, IPython's [Out] output goes to the log file (where, by the convention we have established under Windows, it is not wanted) and not to screen (where it definitely is wanted).
