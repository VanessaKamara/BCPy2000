Future Ideas

  • Factor almost out all the embedded Python code in PythonFilter.cpp into a Python file. Replacement of this Python file would then allow the foundation to call other frameworks besides BCPy2000 more easily (e.g. PyFF).
  • Exploit additional BCI2000 API features
    • OptionalParameter('name', defaultVal)
    • State('name')(offset) = value
  • Implement one or two alternatives to VisionEggRenderer, i.e. graphical systems which are not dependent on OpenGL or VisionEgg.
  • Stimulus composition
    • From PolygonTexture, factor out the code that maps coordinates onto a general-purpose PIL canvas and maps the result onto a VisionEgg TextureStimulus. Have the actual drawing done in a callback between these two steps.
  • Debugging
    • Methods that tell the debugger to stop at a particular event, such as
      • the next transition to a particular state
      • a particular packet number or signal timestamp
    • Keyboard interaction with the source module's stop(), step() and cont(). The developer would need to throw a flag in Construct that tells the superclass to define a "debug" state - then the application module can use it to tell the source module what it has learned from the Event() hook.
