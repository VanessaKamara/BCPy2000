Hook

By hooks we mean methods of a framework object (for example, BciGenericApplication) which you can overshadow when you implement your subclass (e.g. BciApplication). The framework then calls these methods automatically at the appropriate times.

Hooks are identifiable as such by the initial capital letter in their names (and if everything you attach to self, whether it is a method or a data attribute, starts with a capital letter, then you won't clash with anything that already exists). Default implementations (which do nothing) exist for all of them, so you do not have to implement them all. They are:

  • hooks familiar from the BCI2000 C++ framework:
    • Construct called when the module starts
    • Preflight called when "Set Config" is pressed
    • Initialize called directly after Preflight
    • StartRun called when "Start" or "Resume" is pressed
    • Process called on every signal packet
    • StopRun called when the state variable Runnning becomes 0 (perhaps because you have pressed "Suspend")
    • Halt called to clean up the effects of Initialize, when "Set Config" is pressed again
    • Resting called during the time that the system is not running
    • Destruct called when the module closes down
  • an additional hook common to all three BCPy2000 modules:
    • Description allowing you to return a text string (without having to worry about escaping the spaces) describing the module, which will be recorded in the SourceDescription, SignalProcessingDescription or ApplicationDescription parameters, depending on which module you are in.
  • additional hooks for the BCPy2000 application module:
    • Phases to define the temporal and causal structure of the stimulus presentation phases
    • Transition to execute the appropriate actions, and update state variables accordingly, when the presentation phase changes
    • Frame to carry out any operations that must be performed on every video frame
    • Event to process keyboard and mouse events

You can of course implement any further class or instance methods that you might find useful. However, with the exception of the hooks listed above, you should not overshadow existing methods. In particular, you should not have to call, and should definitely not overshadow, any method whose name begins with underscores. Instead of __init__, you should perform setup in Construct (for things that only require to be set up once, independent of parameter settings), Initialize (for setup that needs to be done each time the parameter settings change) and StartRun (for run-specific preparation). Instead of __del__, the corresponding cleanup can be done in Destruct (one time only), Halt (per parameter-setting) and StopRun (per run).
