Command-line Options

Here is a list of command-line options for the application binary:

  • --PythonAppClassFile (default=BciApplication.py): name of the developer file in which you have implemented the BciApplication class
  • --PythonAppWD (default=..\python): where to find the working directory relative to the installation directory
  • --PythonAppShell (default=1): whether to open an interactive shell for this module.
  • --PythonAppLog (no default): the name of a file to which python's sys.stdout and sys.stderr are logged. Currently this excludes the shell's [In] and [Out] traffic -- IPython has its own separate, rather sophisticated logging mechanism. If you want to log something interactively, issue a print statement from the shell prompt. If the characters ### appear in the logfile name, these are replaced by an ISO datestamp.
  • --PythonAppDLL: the name of, or path to, the Python DLL used to create a python engine. By default it looks for python25 first and then python24, but if you have both versions installed, you can specify one or the other explicitly.

The "PythonApp" prefix may seem redundant, but it is necessary because all such command-line options end up being stored in the same file as read-only BCI2000 parameters, and we need to avoid name clashes between the options from different modules. Therefore the signal processing binary uses a different prefix...

  • --PythonSigClassFile (default=BciSignalProcessing.py): name of the developer file in which you have implemented the BciSignalProcessing class
  • --PythonSigWD as above
  • --PythonSigShell as above
  • --PythonSigLog as above
  • --PythonSigDLL as above

...and the signal source binary has yet another prefix:

  • --PythonSrcClassFile (default=BciSource.py): name of the developer file in which you have implemented the BciSource class
  • --PythonSrcWD as above
  • --PythonSrcShell as above
  • --PythonSrcLog as above
  • --PythonSrcDLL as above

If a log file is specified and the shell is disabled, no console window opens. If, furthermore, the log file name is -, nothing is written to disk.

For developers of the BCPy2000 framework itself, there is an additional command-line option --Python*Framework which, if supplied, specifies where the framework files are to be found. Then they are loaded from this location, under the names BCI2000PythonCore.py, BCI2000PythonSource.py, BCI2000PythonSignalProcessing.py and BCI2000PythonApplication.py, instead of from their installed locations.
