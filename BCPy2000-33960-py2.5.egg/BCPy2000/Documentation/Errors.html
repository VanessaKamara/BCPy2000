Errors

If an error message appears in the BCI2000 operator's message window, note that the first word always tells you which module is issuing the message. If the name of the module begins with Python then it is one of the BCPy2000 modules. In this case, the error is from one of three sources:

  1. a serious bug in our C++ code (hopefully now extinct, or at least very rare);
  2. failure to open a required python file (have you installed all the dependencies? have you spelled the name of your developer file correctly?);
  3. an exception that was raised during the execution of python code.

In the case of python exceptions, there are two general kinds:

  1. An exception which is an instance of EndUserError (or a subclass thereof) will deliver its message via the operator and clean up (closing the VisionEgg window, for example). Then, if the error occurred during the Preflight or Initialize stages, the system will wait for the user to adjust the parameters and retry pressing "Set Config". Otherwise it will wait for the user to press "Quit".
  2. Any other kind of exception (whether caused by faulty python code, or deliberately raised by the developer) will issue an operator message that will probably contain the words "Python Error" and/or "BCI2000 must be restarted". Additionally it will cause a colourful IPython backtrace to appear in the corresponding module's console window. The system will not clean up (most objects will not be destroyed, and the VisionEgg window will remain open). The console window will stay open, preventing the module from quitting so that you can read the backtrace. All BCI2000 modules will have to be killed before you can re-try, which will involve you manually closing any console windows in which error backtraces appeared, in addition to quitting from the operator.

So, if the system stops for no apparent reason, check the operator for messages, and check the BCPy2000 console windows for error backtraces. The backtrace should give you the information you need to start fixing the problem (perhaps using pdb, the python debugger). If you are sure that the problem originates in one of the files written by us and want support, copy any backtraces you get so that you can send it with your error report. Copying and pasting from consoles is a little strange under Windows, but the menu from the icon in the top left corner of the console window should enable you to do it.
